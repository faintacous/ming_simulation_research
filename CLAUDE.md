# Ming Simulation Research — 研究协作规范

## 项目定位
- 这是一个**独立的历史研究知识库**，与 Ming_simulation 游戏项目分开管理
- 研究范围：明朝嘉靖年间（1521）至崇祯末年（1644）
- 目标：将散乱的史料整理为结构化的知识成果

## 研究会话流程

每次开新对话研究一个具体主题时，按以下流程进行：

### 第 1 步：明确范围
- 用户指定主题（如"研究：漕运体系"）
- Claude 确认该主题属于哪个目录，是否需要二级细分
- 双方对齐本次会话要产出什么

### 第 2 步：搜集与整理
- Claude 使用 WebSearch 搜集公开资料，整理出初稿
- 用户对照自身知识补充或纠正
- 标注信息来源

### 第 3 步：结构化归档
- 按下方文档模板产出标准格式文件
- 存入对应主题目录
- 更新来源追踪表（`sources.md`）

### 第 4 步：标记关联与存疑
- 标记与其他主题的关联点（如"漕运损耗 → 04-赋税与财政/折银动机"）
- 列出存疑点和待查内容，供后续会话跟进

## 工作原则

1. **研究与游戏项目分离** — 这里只做史料整理，不涉及游戏设计决策
2. **每次会话聚焦一个具体子主题** — 不在一个对话里混合多个主题
3. **用户把关史实判断** — Claude 负责搜集和结构化，用户负责验证准确性
4. **来源必须标注** — 每条关键信息都要标明出处
5. **存疑要明确标出** — 不确定的内容不混入"关键事实"，单独列入存疑区

6. **本项目采用高度自主化工作模式** — 方向对齐后，细节决策不需要反复请示
   - 需要请示：(a) 任务范围变更；(b) 不可逆破坏性操作（删除历史数据、force push、覆盖无副本文件）；(c) 方向性选择（A 路线 vs B 路线对质量目标影响较大）；(d) 质量目标不清时的优先级锚定
   - 不需要请示：流程参数（超时/并发度/重试次数）、字段措辞、实现细节（库选择/命名/日志格式/输出位置）、执行时机（后台/前台/衔接时机）、**优化迭代的启动**（发现质量不达目标时自主启动迭代）、**过程记录文档的创建**（写在 `docs/` 下的工程记录不需要批准）
   - 汇报节奏：关键里程碑（阶段完成、质量交付、重大方向转换）一定汇报；日常进度不汇报；汇报偏"已完成 + 下一步"，不偏"我想做 X 可以吗"
   - 与全局规则 R1「写代码之前先说清楚」的关系：R1 指方案级对齐（用户有知情权），不是细节确认。本条是 R1 在本项目的适用边界细化

## 目录与文件管理规则

为防止目录膨胀和临时文件堆积，所有文件创建遵循以下约束：

1. **顶层目录只能属于「主题目录」或「基础设施目录」两类**
   - 主题目录：本文件「主题目录」表中列出的 12 个研究分类，以及已约定的工作目录（`人物名录/`、`角色视角/`、`专题研究/`、`文献收集/`、`docs/`、`tools/`、`logs/`、`memory/`、`.claude/`）
   - 新开顶层目录**必须先在 CLAUDE.md 注册**，否则视为越界
   - 禁止创建职能重叠的并行目录（如「人物名录 / 人物传记 / 人物档案」三开）

2. **脚本一律放 `tools/`，不散落根目录**
   - 根目录的 `*.py` `*.sh` 只保留入口级调度脚本（当前无此类），其他全部归入 `tools/`

3. **脚本迭代用 git 而不是文件名**
   - 不允许 `xxx_v2.py`、`xxx_new.py`、`xxx_rerun.py` 等版本后缀并存
   - 重大重写时旧版本**直接删除**，历史由 git 承载
   - 例外：正在 A/B 对比期间可短期并存，对比结束后立即合并或删除

4. **临时 / 调试文件不落根目录**
   - 一次性产物放 `tmp/`（已在 `.gitignore`），或走管道不落盘
   - 调试片段写完即删，不要作为「中间版本」留存
   - 日志统一进 `logs/`

5. **备份目录有保质期**
   - 备份命名须带日期与用途：`_backup_{用途}_{YYYYMMDD}/`
   - 主文件稳定 7 天无回滚需求后**必须删除**对应备份
   - 长期参考价值的快照改用 git tag，不用目录留存

6. **新增文件/目录前自检三问**
   - 是否已有同职能位置可以放？
   - 是否会和现有目录形成职能重叠？
   - 是否只是本次会话的临时产物（若是，进 `tmp/` 或不落盘）？

7. **批量删除前必须白名单交叉校验**
   - 凡涉及 `rm -r`、批量清理未跟踪文件等动作，执行前必须先打印待删清单
   - 清单与「本任务要保留的文件」逐项比对，确认无交集再执行
   - 未跟踪 ≠ 可删；「git status 显示 `??`」只说明未纳入版本控制，不说明它是临时产物

## 多 Agent 协作研究规范

本项目采用多 Agent 流水线进行大规模研究，由 `tools/research_pipeline.py` 和 `tools/batch_research.py` 调度。以下规范约束各 Agent 的职责、产出标准和协作流程。本章节经过 2026-04-09 至 04-12 期间人物名录 67 篇传记全库校验的实战检验（见 `人物名录/_validation/综合校验报告.md`、`全库工单.md`、`联网核对报告.md`、`BIO-006验收报告.md`、`交叉一致性报告.md`），对各模型的能力边界做了明确的实证校准。

### Agent 角色定义

| Agent | 定位 | 核心职责 | 适用场景 | 不适用场景 |
|-------|------|----------|----------|------------|
| **Kimi** | 广度搜集者 | WebFetch + WebSearch 综合多源，产出 300–650 行长稿 | 新传记初稿、主题首次搜集、需要案例+制度+数据三层论证的长文 | 精细事实核对、跨文件一致性检查、超过日配额后的持续调用 |
| **Codex** | 深度校验者 | WebFetch 访问 wikisource/ctext/CBDB、跨文件分析、结构化工单产出 | 内部矛盾检测、跨文件一致性检查（BIO-005）、联网核对一手史料、诊断单/工单生成 | 大规模新内容生成（单次产出通常 30–100 行，不是它的长项） |
| **opencode + glm-5.1** | 生成式研究代理（备用） | WebFetch、Write、Bash，可按诊断单执行批量修订 | Kimi 日配额耗尽后的备用生成器、按既定工单的批量修订 | 无监督批量生成（易错，质量不如 Kimi）、精确文件路径操作 |
| **Opus** | 裁决合并者 | 读取多份草稿，裁决冲突，合并为符合文档模板的终稿 | 综合判断、格式统一、冲突仲裁 | 独立搜集新资料（依赖上游草稿质量） |
| **用户** | 最终把关者 | 验证史实准确性，补充领域知识，审批终稿 | 领域判断、最终决策 | — |

### 工具能力边界（实战验证）

本小节记录 2026-04 全库校验过程中对三个模型实际能力的观测结果，供后续调度决策参考。

#### Kimi

- **配额是 5 小时滚动窗口限流，窗口内约 3-10 次 step 可能打穿**（原粗估"约 17–18 次"仅为峰值观测，非日配额；实际是 5h 滚动窗口）。超过后触发 `rate_limit_reached_error`，需等窗口滑出才能继续。规划批处理任务时必须按"5h 窗口"预留配额，或拆分到跨窗口执行。
- **擅长长稿综合**：单次产出稳定在 300–650 行，结构完整，案例丰富。
- **弱点**：对"已有文件里的具体一句话是否有原文支撑"这类精细核对任务不擅长；对跨文件矛盾检测没有稳定能力。

- **日配额观测修正（2026-04-16）**：原 CLAUDE.md 写"约 17–18 次"为粗估；实测 A 线 3 人 Kimi 重建成功后，再铺 10 人 3 并发代理即触发 429。实际可触发因素：
  - Kimi Agent 内部 Step 分裂：单次人工感知的"调用"可能在后端消耗 2–5 次
  - 并发代理叠加账号级配额（并发无加速，反而快速打穿）
  - **新规则**：Kimi 建档任务**单代理串行**，不并发多代理；铺量前主窗口先做 1 次 ping 探测（一个简短 kimi 调用看是否 429），推荐使用 `tools/kimi_safe_call.sh --ping` 或 `tools/kimi_safe_call.py --ping`

- **Windows 平台调用规范（2026-04-16）**：
  - 环境变量必设：`PYTHONIOENCODING=utf-8` + `PYTHONUTF8=1`
  - kimi-cli rich 渲染字符（• ◄ ► 等）在 GBK 控制台会 `UnicodeEncodeError: 'gbk' codec can't encode` 崩溃
  - 优先用 `tools/research_pipeline.py` 的 `run_kimi()` 接口（已处理 GBK）或本次沉淀的 `tools/kimi_safe_call.sh` / `tools/kimi_safe_call.py` 包装
  - 让 Kimi 自写 .md 到工作目录，而不是 stdout pipe 给 tee（stdout 含 rich 字符，pipe 到 tee/GBK 终端必崩）

- **跨工具 `/tmp` 路径陷阱（2026-04-16 发现）**：
  - Git Bash 的 `/tmp` → `E:\System_Temp\`（Cygwin/MSYS mount）
  - Claude Code Write 工具的 `/tmp` → `G:\tmp\`（相对根盘）
  - Python subprocess 起的 `/tmp` → `G:\tmp\`（Windows 解释）
  - 三者解释不同字符串 → 跨工具协作时 `/tmp/xxx.md` 指向三个不同位置，导致 `FileNotFoundError`
  - **绝对禁用 `/tmp` 字符串跨工具传递**；临时文件统一放项目内 `tmp/` 目录（相对 `G:/AIDev/Ming_Simulation_Research/`）；代理间传递路径必须用绝对路径或项目相对路径

- **`400 content_filter` 是独立于 429 的硬拒机制（2026-04-16）**：触发词是"斩/构陷/奏杀/阉党/下狱论死"等政治暴力密度高的动词组合。应对：用"本传客观叙事"风格写 prompt，避免评价性动词和情绪化用语。钱龙锡 prompt 连拒 3 次为此教训代价；李待问风格即为合规范本。

- **work_dir salvage 兜底是刚需（2026-04-16）**：Kimi 在 429/content_filter/timeout 前常已写出完整或大段稿。`kimi_safe_call.py` 的 tempdir 产物不应删除，driver 必须在所有失败分支调用 salvage 函数。张经 659 行、夏允彝 360 行完整稿均由此机制救回。

- **幂等 reuse out 机制（2026-04-16）**：driver kill/重启场景下，已有 out 产物应直接搬运到 target，跳过 Kimi 调用（节省窗口配额）。李待问首次执行即用 0 秒完成归档。

#### Codex

- **目前未见配额上限**，可作为常驻校验工具使用。
- **WebFetch 能稳定访问** `zh.wikisource.org/wiki/明史/卷XXX`、`ctext.org/datawiki.pl`、`cbdb.fas.harvard.edu/cbdbapi/person.php` 等一手史料站点，并能按"打开的实际页面"而非训练记忆作答（见 `联网核对报告.md` 的原文锚定风格）。
- **产出短小但密度高**：单次调用通常 30–100 行，擅长"给一张清单而不是写一篇长文"。
- **跨文件能力强**：能识别多篇文档间的同一事件叙述差异（BIO-005 类问题），是唯一在本项目中可靠完成交叉一致性检查的工具。

#### opencode + glm-5.1

- **配置陷阱**：默认 `model` 是 `glm-4.6`，存在严重的 SSE 超时问题。必须在 agent 配置文件里显式指定 `model: zhipuai-coding-plan/glm-5.1`，不要依赖默认值。
- **工具禁用的边界**：工具禁用必须精确到具体工具名。`question` 是模型的内置能力，无法通过 agent 白名单关掉，只能在 prompt 设计上消除其触发条件。
- **路径遵从性差**：模型不严格遵守精确文件路径指令，常写到错误位置。应对办法是用脚本自动搬运产物到目标目录，不要在 prompt 里加码对抗。
- **适合场景**：Kimi 配额耗尽后的备用生成器、按 Codex 已生成的诊断单做批量修订。不适合无监督的大批量创建任务。

### 推荐工作流（基于三日实战总结）

根据全库校验的实际执行经验，推荐以下三种标准流水线：

#### 工作流选择决策树（必读——2026-04-15 P1 批次教训）

**执行任何传记任务前，先判断目标文件状态**：

| 目标文件状态 | 正确工作流 | 错误陷阱 |
|-------------|-----------|---------|
| 不存在 / 空壳 / 仅有标题 | 工作流 A（Kimi 初稿） | — |
| 已存在且有实质内容，但质量有问题 | **工作流 C（Codex 诊断 → 按单修订）** | 不要走 B 让 Kimi 重写整篇 |
| 已存在且质量合格，仅需扩展章节 | 在现有基础上追加章节（可局部 Kimi 补写），**不整篇覆盖** | 不要整篇 Kimi 重写 |

**为什么不能对已有传记用 Kimi 整篇重写**：2026-04-15 P1 批次（6 人：张居正/海瑞/朱翊钧/王喜姐/郑贵妃/李时珍）实测结果——

- Kimi 重写覆盖 → Codex 三阶段校验 → BIO-005/006 机械审计全部通过
- 但 Codex 新旧质量对比评审显示：2/6 显著提升、3/6 退化、1/6 轻微退化
- 一手古籍来源数平均 **-17.6%**（核心负面信号）
- 张居正 5 条 04-12 已修旧错被 Kimi 重写**回滚**
- 朱翊钧新版引入腾讯新闻/知乎/网易号等低学术源
- 郑贵妃"大高玄殿盟誓"从存疑区**重新爬回正文**

**根因**：Kimi 重写产生的是全新文本，不保证保留旧版已修复的事实与来源。BIO-006 只校合规性，不校是否比旧版更好。

**Commit 前拦截**：对已有传记的任何修订，commit 前**必须**执行"新旧质量评审"——Codex 对比 `git HEAD` 旧版与当前新版，按以下维度打分：
- 一手古籍独立来源数（旧/新）
- 《明史》卷次引用数（旧/新）
- 04-12（或上轮）校验报告标记的具体问题是否仍在
- 是否引入新低学术源

**拦截阈值**：一手来源数下降超过 15%、或 04-12 已修问题被回滚、或出现新低学术源 → **禁止 commit**，必须先做"旧版优质内容合并"补救（参考 2026-04-15 P1 批次 `_validation/P1_批次_20260415/` 下的合并产出模板）。

- **《明史》独立卷号去重数下降**：若单人 HEAD 独立卷号 ≥ 3 但新版 ≤ 1，视同来源单薄化，拦截（2026-04-16 BIO-002 重扫发现张居正 58 次引用挤在卷 213 一个卷号，提示"一手引用总数 ≠ 来源多样性"；评审阈值必须同时看"引用次数"和"独立卷号去重数"两个维度）

**补救规则**（旧→新合并时的硬规则）：
- 应补：带卷次的一手古籍引用、前轮校验已修复的事实（官职/年份/人名订正）、准确日期/人际关系
- 应舍：无卷次单百科、野史孤证、腾讯新闻/知乎/网易号、与修订后文本冲突且缺一手支撑的叙述

#### 工作流 A：新传记建档

```
Kimi 搜集初稿（长稿 300–650 行） → 存盘为 {人物}.md
```

新条目从零生成时，首选 Kimi 单轮产出，不要在建档阶段混入其他工具。

#### 工作流 B：新传记校验（完整版）

```
Codex 综合校验（找矛盾、产出 BIO-00x 工单）
  ↓
按工单修订（人工 or Agent 按单执行）
  ↓
Codex 联网核对一手史料（对照 wikisource/ctext/CBDB 原文）
  ↓
Codex 跨文件一致性检查（BIO-005：同事件在不同条目是否一致）
  ↓
BIO-006 验收脚本（_validation/_audit_script.py 自动扫描 R1–R6 六条规则）
```

### 六个关键教训（陷阱提示）

以下是 2026-04 校验过程中付出调试成本后沉淀的六条硬规则，后续任何涉及多模型调度的场景都必须先过一遍这个清单：

1. **禁用工具要精确到具体工具名，不要一刀切**——粗粒度禁用会误伤有用能力，细粒度禁用才能既防风险又保留价值。
2. **配置文件必须显式声明 model，不依赖默认值**——默认模型可能有超时或质量问题（如 glm-4.6 的 SSE 超时），显式声明是唯一可靠方式。
3. **模型不听指令时用脚本搬运，不要加码对抗**——反复在 prompt 里强调"写到 X 路径"是无效投入，自动化搬运脚本才是根治方案。
4. **某些模型能力（如 question）禁不掉，只能消除触发条件**——遇到这类能力时停止尝试禁用，改从 prompt 结构入手避免触发。
5. **单文件修订有盲区，必须有 BIO-005 跨文件检查**——即使每篇单独校验都合格，跨文件的同事件叙述仍可能互相冲突（如梃击案、高拱被逐、李成梁收养说）。
6. **用工具之前先真正测试能力边界，不要凭想当然**——每个模型的配额、工具权限、响应质量都要实测确认，不要套用其他项目的经验。

### 草稿产出标准

#### Kimi 草稿要求
- **篇幅**：400–700 行
- **结构**：以个体完整经历为主线，从"如何进入"到"最终结局"
- **必含要素**：
  - 每个阶段的制度性规定（引用法典/会典条文）
  - 具体数据（人数、银两、比例等，标注来源）
  - 真实案例（至少 3 个有名有姓的历史人物）
- **来源标注**：古籍须注明卷次（如《明史·职官志》卷七十二）
- **输出文件**：`角色视角/{topic}_kimi.md`

#### Codex 草稿要求
- **聚焦**：一手古籍的具体卷次和条文引证
- **必含要素**：
  - 职业生命周期模型（分阶段框架）
  - 核心研究问题提炼（3–5 个可验证问题）
  - 分期观察（嘉靖 / 隆万 / 天启崇祯的变化趋势）
  - 存疑与待查清单（对无法确认的数据标注"存疑"）
- **输出文件**：`角色视角/{topic}_codex.md`

### 质量门槛（进入合并的准入条件）

草稿必须满足以下条件才能进入 Opus 合并阶段：

| 条件 | Kimi | Codex |
|------|------|-------|
| 最低行数 | ≥ 200 行 | ≥ 50 行 |
| 来源标注 | 每条关键事实都有出处 | 古籍引用须有卷次 |
| 时段覆盖 | 至少覆盖嘉靖至崇祯中的两个时段 | 提供分期观察 |
| 结构完整 | 含入行、日常、收入、风险、结局五段 | 含生命周期模型 |

**不满足门槛时**：Opus 在合并报告中标注缺失项，由用户决定是重跑还是人工补充。

### 冲突仲裁规则

当 Kimi 和 Codex 的信息不一致时，Opus 按以下优先级裁决：

1. **一手古籍 > 二手转述** — 直接引用《明史》《大明会典》《大明律》原文的版本优先
2. **有卷次 > 无卷次** — 标注了具体卷次/条文的引用优先
3. **学术共识 > 孤证** — 多个独立来源支持的结论优先
4. **两者均有理据时** — 保留两说并列，标注"待用户裁决"，移入存疑区
5. **数据冲突** — 列出双方数据及来源，不做取舍，交由用户判断

### Opus 合并规范

#### 合并流程
1. 以 Kimi 草稿为主体框架（占终稿 70–85%）
2. 将 Codex 的分析框架、补充引证、存疑清单整合到对应章节
3. Codex 独有的生命周期模型和核心问题，作为"校验稿补充"附于文末
4. 统一格式，确保终稿符合本文件"文档模板"章节的结构要求
5. 生成合并报告，列出：采纳了什么、丢弃了什么、存疑有哪些

#### 终稿输出
- **文件**：`角色视角/{topic}.md`
- **格式**：必须包含文档模板中的所有章节（概述、关键事实、数据、时间线、关联主题、存疑、来源汇总）
- **合并完成后**：删除 `_kimi.md` 和 `_codex.md` 草稿文件（已归档到终稿中）
