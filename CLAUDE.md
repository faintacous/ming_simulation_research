# Ming Simulation Research — 研究协作规范

## 项目定位
- 这是一个**独立的历史研究知识库**，与 Ming_simulation 游戏项目分开管理
- 研究范围：明朝嘉靖年间（1521）至崇祯末年（1644）
- 目标：将散乱的史料整理为结构化的知识成果

## 研究会话流程

每次开新对话研究一个具体主题时，按以下流程进行：

### 第 1 步：明确范围
- 用户指定主题（如"研究：漕运体系"）
- Claude 确认该主题属于哪个目录，是否需要二级细分
- 双方对齐本次会话要产出什么

### 第 2 步：搜集与整理
- Claude 使用 WebSearch 搜集公开资料，整理出初稿
- 用户对照自身知识补充或纠正
- 标注信息来源

### 第 3 步：结构化归档
- 按下方文档模板产出标准格式文件
- 存入对应主题目录
- 更新来源追踪表（`sources.md`）

### 第 4 步：标记关联与存疑
- 标记与其他主题的关联点（如"漕运损耗 → 04-赋税与财政/折银动机"）
- 列出存疑点和待查内容，供后续会话跟进

## 文档模板

每个研究主题产出的文档遵循以下格式：

```markdown
# [主题名称]

## 概述
（200字以内的总结）

## 关键事实
- 事实1 [来源：xxx]
- 事实2 [来源：xxx]

## 数据（如适用）
| 维度 | 数据 | 来源 |
|------|------|------|

## 时间线（如适用）
- 年份：事件 [来源]

## 关联主题
- → [目录/主题名]：关联说明

## 存疑 & 待查
- 问题1
- 问题2

## 来源汇总
1. 来源名称 — 类型（古籍/论文/百科/专著）— 可信度评估
```

## 工作原则

1. **研究与游戏项目分离** — 这里只做史料整理，不涉及游戏设计决策
2. **每次会话聚焦一个具体子主题** — 不在一个对话里混合多个主题
3. **用户把关史实判断** — Claude 负责搜集和结构化，用户负责验证准确性
4. **来源必须标注** — 每条关键信息都要标明出处
5. **存疑要明确标出** — 不确定的内容不混入"关键事实"，单独列入存疑区

## 多 Agent 协作研究规范

本项目采用多 Agent 流水线进行大规模研究，由 `research_pipeline.py` 和 `batch_research.py` 调度。以下规范约束各 Agent 的职责、产出标准和协作流程。

### Agent 角色定义

| Agent | 定位 | 核心职责 | 优势 | 局限 |
|-------|------|----------|------|------|
| **Kimi** | 广度搜集者 | 搜集史料，构建完整叙事，提供制度+数据+案例三层论证 | 信息密度高、案例丰富、可读性强 | 可能引用未经严格校验的二手资料 |
| **Codex** | 深度校验者 | 交叉校验，聚焦一手古籍引用，提出分析框架与待验证假设 | 强调原始文献卷次、分析框架新颖 | 产出篇幅较短，覆盖面有限 |
| **Opus** | 裁决合并者 | 读取两份草稿，裁决冲突，合并为符合文档模板的终稿 | 综合判断力强，可统一格式和逻辑 | 依赖草稿质量，不独立搜集新资料 |
| **用户** | 最终把关者 | 验证史实准确性，补充领域知识，审批终稿 | 领域判断力，最终决策权 | — |

### 草稿产出标准

#### Kimi 草稿要求
- **篇幅**：400–700 行
- **结构**：以个体完整经历为主线，从"如何进入"到"最终结局"
- **必含要素**：
  - 每个阶段的制度性规定（引用法典/会典条文）
  - 具体数据（人数、银两、比例等，标注来源）
  - 真实案例（至少 3 个有名有姓的历史人物）
- **来源标注**：古籍须注明卷次（如《明史·职官志》卷七十二）
- **输出文件**：`角色视角/{topic}_kimi.md`

#### Codex 草稿要求
- **聚焦**：一手古籍的具体卷次和条文引证
- **必含要素**：
  - 职业生命周期模型（分阶段框架）
  - 核心研究问题提炼（3–5 个可验证问题）
  - 分期观察（嘉靖 / 隆万 / 天启崇祯的变化趋势）
  - 存疑与待查清单（对无法确认的数据标注"存疑"）
- **输出文件**：`角色视角/{topic}_codex.md`

### 质量门槛（进入合并的准入条件）

草稿必须满足以下条件才能进入 Opus 合并阶段：

| 条件 | Kimi | Codex |
|------|------|-------|
| 最低行数 | ≥ 200 行 | ≥ 50 行 |
| 来源标注 | 每条关键事实都有出处 | 古籍引用须有卷次 |
| 时段覆盖 | 至少覆盖嘉靖至崇祯中的两个时段 | 提供分期观察 |
| 结构完整 | 含入行、日常、收入、风险、结局五段 | 含生命周期模型 |

**不满足门槛时**：Opus 在合并报告中标注缺失项，由用户决定是重跑还是人工补充。

### 冲突仲裁规则

当 Kimi 和 Codex 的信息不一致时，Opus 按以下优先级裁决：

1. **一手古籍 > 二手转述** — 直接引用《明史》《大明会典》《大明律》原文的版本优先
2. **有卷次 > 无卷次** — 标注了具体卷次/条文的引用优先
3. **学术共识 > 孤证** — 多个独立来源支持的结论优先
4. **两者均有理据时** — 保留两说并列，标注"待用户裁决"，移入存疑区
5. **数据冲突** — 列出双方数据及来源，不做取舍，交由用户判断

### Opus 合并规范

#### 合并流程
1. 以 Kimi 草稿为主体框架（占终稿 70–85%）
2. 将 Codex 的分析框架、补充引证、存疑清单整合到对应章节
3. Codex 独有的生命周期模型和核心问题，作为"校验稿补充"附于文末
4. 统一格式，确保终稿符合本文件"文档模板"章节的结构要求
5. 生成合并报告，列出：采纳了什么、丢弃了什么、存疑有哪些

#### 终稿输出
- **文件**：`角色视角/{topic}.md`
- **格式**：必须包含文档模板中的所有章节（概述、关键事实、数据、时间线、关联主题、存疑、来源汇总）
- **合并完成后**：删除 `_kimi.md` 和 `_codex.md` 草稿文件（已归档到终稿中）

### 用户介入点

以下环节需要用户确认后才能继续：

| 环节 | 用户动作 |
|------|----------|
| 主题选定 | 确认研究范围和预期产出 |
| 冲突裁决 | 对"待用户裁决"的条目做出判断 |
| 终稿审批 | 审阅合并后的终稿，标记需修改的部分 |
| 存疑处理 | 决定哪些存疑需要后续深挖，哪些可以搁置 |

**自动化环节**（无需用户介入）：Kimi/Codex 并行搜集、草稿质量门槛检查、格式统一。

### 会话间上下文传递

- 每次合并完成后，更新 `sources.md` 来源追踪表
- 存疑条目带上来源会话标记，便于后续会话定向跟进
- 关联主题的交叉引用使用格式：`→ [目录/主题名]：关联说明`
