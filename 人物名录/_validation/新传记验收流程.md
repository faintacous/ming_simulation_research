# 新传记验收流程（操作手册）

本文档说明：**每次新增或修订一份人物传记后，应该跑哪些验收命令、
检查什么、通过标准是什么、不通过如何处理。**

工具实现位于 [`_validation/tools/`](tools/)，详细用法见
[`tools/README.md`](tools/README.md)。

## 前置条件

1. 已完成 `研究会话流程` 的第 1–3 步：范围确认、搜集整理、结构化归档。
2. 多 Agent 流水线（Kimi + Codex → Opus）已产出合并终稿，草稿文件
   （`_kimi.md` / `_codex.md`）已删除。
3. 终稿按照 CLAUDE.md 的**文档模板**书写，章节顺序为：
   - 基本信息 → 生平概要 → 关键事迹 → 争议与待考 → 来源汇总 → 关联主题

## 标准验收流程

### 第 1 步：全量跑一次验收

在仓库根目录执行：

```bash
python 人物名录/_validation/tools/run_validation.py \
    --base-dir 人物名录 --all
```

输出示例：

```
--- 运行 BIO-001 ---
BIO-001: 65/67 通过
--- 运行 BIO-003 ---
BIO-003: 3 文件命中，共 5 条违规
--- 运行 BIO-006 ---
BIO-006: 66/67 非严重不合规

=== 总结 ===
有 2/3 项规则未通过，请查看上方报告。
```

对应报告：
- `_validation/BIO-001结构检查报告.md`
- `_validation/BIO-003扫描报告.md`
- `_validation/BIO-006验收报告.md`
- `_validation/BIO-006审计结果.json`（机器可读，便于后续聚合）

### 第 2 步：按规则查看问题

#### BIO-001：结构检查

**检查什么**：
- 必备章节（5 个）是否齐备：基本信息、生平概要、关键事迹、
  争议与待考、来源汇总；
- 主干四段（基本信息 → 生平概要 → 关键事迹 → 争议与待考）顺序是否正确；
- "来源汇总"和"关联主题"是否都排在主干之后；
- "基本信息"是否为第一个二级章节。

**通过标准**：
- **通过**：必备章节齐备、顺序正确、基本信息在首位；
- **warning**：可选章节"关联主题"缺失（不影响通过，但建议补上）；
- **不通过**：以上任一硬性要求不满足。

**不通过如何处理**：
1. 打开报告 `_validation/BIO-001结构检查报告.md`，找到本次新增的文件；
2. 根据"缺失必备章节"或"顺序错误"提示，回到传记文件调整章节结构；
3. 重跑 `--bio001`，直到通过。

#### BIO-003：争议下沉扫描

**检查什么**：正文中是否存在"存疑 / 待考 / 原始出处待查 / 来源：存疑 /
野史 / 民间传说"等字样散落在"存疑 & 待查"章节之外。

**通过标准**：零命中（`BIO-003: 0 文件命中`）。

**不通过如何处理**：
1. 打开报告 `_validation/BIO-003扫描报告.md`，找到本次新增文件对应的逐行清单；
2. 对每一条命中做判断：
   - 如果是**争议叙述**（需要下沉）：把整句搬运到"## 存疑 & 待查"章节；
   - 如果是**元数据/分级标签**（不应算违规）：考虑为该行加入
     `tools/bio003_dispute_scan.py` 的噪声模式清单；
   - 如果是**合规的来源分级标签**（如"民间传说与地方记忆"作为来源
     等级标题）：无需处理，脚本默认已过滤。
3. 对已彻底修订的文件，可以使用 `--exclude-file` 在后续扫描中跳过：

```bash
# 创建已修订清单
echo "新增人物.md" >> _validation/已修订清单.txt

python 人物名录/_validation/tools/run_validation.py \
    --base-dir 人物名录 --bio003 \
    --exclude-file _validation/已修订清单.txt
```

#### BIO-006：全库验收审计

**检查什么**（六条规则）：

| 规则 | 标准 | 新传记应达到 |
|------|------|--------------|
| R1 一手依据 | ≥ 2 条一手或高等级二手依据 | 必须满足 |
| R2 存疑搬运 | 正文无 `[来源：存疑]` 类句内标注 | 必须满足 |
| R3 未决争议清单 | 有"存疑 & 待查"章节且含实质条目 | 必须满足 |
| R4 来源汇总 | ≥ 5 条条目 | 必须满足 |
| R5 基本信息完整 | 生卒/字号/籍贯/身份 ≥ 3 项 | 必须满足 |
| R6 文件长度 | ≥ 100 行 | 必须满足；建议 ≥ 200 行 |

**通过标准**：
- **全合规**：6 条规则全过且长度 ≥ 200 行；
- **合格偏短**：6 条规则全过，长度 100–200 行（可接受，建议扩写）；
- **部分不合规**：1–2 条规则失败，且不涉及 R1/R2（需要修订）；
- **严重不合规**：R1 或 R2 失败，或 ≥ 3 条规则失败（必须修订）。

**对新传记的硬性要求**：**必须达到"全合规"或"合格偏短"**；
"部分不合规"和"严重不合规"不得纳入正式仓库。

**不通过如何处理**：

1. **R1 失败**（一手依据不足）：
   - 回到多 Agent 流水线，要求 Codex 补充更多一手古籍卷次引用；
   - 或人工补充 `《明史》卷 X`、`《明实录》某年某月` 等出处。
2. **R2 失败**（正文残留存疑标注）：
   - 执行 BIO-003 下沉动作（见上一节）。
3. **R3 失败**（缺存疑章节或为空）：
   - 即使全部内容都已核实，也应保留一个写有"暂无未决争议"的空壳
     章节，以示已做过存疑评估。
4. **R4 失败**（来源汇总不足）：
   - 补足至少 5 条条目；
   - 支持列表格式（`- 来源名 — 类型 — 可信度`）或表格格式。
5. **R5 失败**（基本信息缺失）：
   - 补充缺失字段；未知的字段可写"不详"或"约 XXXX 年"。
6. **R6 失败**（文件过短）：
   - < 100 行的文件基本等同"未完成"，应回到 Kimi/Codex 流水线重跑。

### 第 3 步：修订后重跑

完成修订后重跑全量验收：

```bash
python 人物名录/_validation/tools/run_validation.py \
    --base-dir 人物名录 --all
```

**最终通过标准**：
- BIO-001：本次新增文件在"通过"清单中（可有 warning）；
- BIO-003：本次新增文件不在"逐文件清单"中；
- BIO-006：本次新增文件被归类为"全合规"或"合格偏短"。

上述三条全部满足后，新传记才可以纳入仓库。

### 第 4 步：更新追踪信息

验收通过后：

1. 更新 [`sources.md`](../sources.md) 来源追踪表（新增文件的来源条目）；
2. 如果存在存疑条目需要后续深挖，在会话记忆中打上来源会话标记；
3. 如果与其他条目存在交叉关联，使用 `→ [目录/主题名]：关联说明` 格式
   添加到"关联主题"章节。

## 批量验收（用于定期盘点）

除了单份新增，也可以定期对全库跑一次批量验收，评估整体合规率变化：

```bash
# 1. 跑全量
python 人物名录/_validation/tools/run_validation.py \
    --base-dir 人物名录 --all

# 2. 查看三份报告，聚焦新出现的不合规项
# 3. 如果合规率下降，排查是否有人绕过验收直接提交
```

建议频率：每完成一组（例如一个目录级的 BIO-20x 工单）后跑一次。

## 常见误判与处理

### 1. BIO-003 误报来源分级标题

如果某份传记的"来源汇总"章节使用了 `#### — 野史 —` 或
`#### 民间传说与地方记忆` 作为子标题（来源分级标签），默认噪声模式
已覆盖。如果仍被误报，编辑 `tools/bio003_dispute_scan.py` 的
`NOISE_PATTERNS` 清单即可。

### 2. BIO-006 R1 误判

`R1` 通过关键词命中兜底，如果某篇用的是不常见一手史料（如罕见的族谱、
海外手稿），脚本可能识别不到。此时可在 `tools/bio006_audit.py` 的
`PRIMARY_KEYWORDS` 清单里加上该史料名。

### 3. BIO-001 章节别名不匹配

如果新传记使用了一种未被别名清单覆盖的章节名（例如用了"家世与早年"
代替"生平概要"），脚本会报"缺失必备章节"。**优先改传记用标准章节名**，
而不是在脚本里加新别名——保持章节命名一致性本身就是 BIO-001 的目的。

## 工具维护

工具本身也是可迭代的。发现以下情况时，应更新工具而不是绕过它：

- 反复出现相同的误判 → 调整过滤规则；
- 新的史料来源类型频繁出现 → 扩充关键词清单；
- 新的章节命名习惯 → 评估是收敛命名还是扩充别名；
- BIO-002 / BIO-004 / BIO-005 找到机器可判据 → 在 `run_validation.py`
  的 `RULES` 注册表中添加新规则。
