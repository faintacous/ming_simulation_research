## 一、12 步建档批次全链路

每次"批次建档"指一次同主题/同区域 N 人传记从无到有的完整流程。下表覆盖最理想路径；实战会因文件状态分流（见第四章决策点）。

| # | 阶段 | 动作 | 工具 | 主产物 | 路径与命名 |
|---|------|------|------|--------|-----------|
| 1 | 候选清单 | 圈定批次人选与建档形式（单/合/名） | 主窗口 + Codex 调研 Agent | 候选清单 | `人物名录/_候选清单/{批次}.md` 或 INDEX 标注 |
| 2 | Prompt 准备 | 为每人写定 Kimi prompt（客观叙事体） | 主窗口 Write | prompt 文件 | `tmp/{批次}_prompts/{topic}_prompt.txt` |
| 3 | Kimi ping | 探测 5h 窗口是否可用 | `tools/kimi_safe_call.py --ping` | stderr 信号 | — |
| 4 | Kimi 建档 | 单代理串行调用，不并发 | `tools/kimi_safe_call.py` | 长稿 .md | `tmp/{批次}_out/{topic}.md` |
| 5 | salvage 兜底 | 失败分支救回 work_dir 产物 | driver 内置或手动 | 部分稿 .md | 同上 |
| 6 | 归档主库 | 把 Kimi 产物搬运到正式位置 | 主窗口 Bash cp/mv | 终稿 | `人物名录/{分类}/{人名}.md` |
| 7 | 综合校验 | Codex 找内部矛盾、产工单 | `codex exec` | 校验报告 | `_validation/{批次目录}/{批}_{topic}_综合校验_{日期}.md` |
| 8 | 联网核对 | Codex WebFetch wikisource/ctext/CBDB | `codex exec` (read-only) | 核对报告 | `_validation/{批次目录}/{批}_{topic}_联网核对_{日期}.md` |
| 9 | 跨文件一致性 | BIO-005：同事件多传记互校 | `codex exec` | BIO-005 报告 | `_validation/{批次目录}/{批}_BIO005_跨文件一致性_{日期}.md` |
| 10 | 机械审计 | BIO-006/008/009 脚本扫描 | `tools/bio*.py` + `_validation/tools/` | json + md | `_validation/_全库审计/` 或批次目录 |
| 11 | 修订执行 | 按工单做 Edit | 主窗口或 Codex Agent | 编辑执行报告 | `_validation/{批次目录}/{批}_{topic}_编辑执行报告_{日期}.md` |
| 12 | 新旧评审 | Codex 对比 git HEAD vs 当前 | `codex exec` | v3 评审报告 | `_validation/{批次目录}/{批}_{topic}_新旧质量评审_v{N}_{日期}.md` |

**commit 时机**：第 12 步通过四条拦截阈值（见第五章）后才能 commit。

## 二、关键工具与接口

### Kimi（建档生成）

**入口**：`tools/kimi_safe_call.py`

```bash
# 探测（铺量前必跑）
python tools/kimi_safe_call.py --ping

# 调用
python tools/kimi_safe_call.py \
    --prompt-file tmp/songjiang3_prompts/songzhengyu_prompt.txt \
    --out-file tmp/songjiang3_out/songzhengyu.md \
    --topic songzhengyu \
    --timeout 1800
```

**退出码**：`0` 成功 / `429` 配额耗尽 / `124` 超时 / `1` 其他错误

**硬约束**：
- **5h 滚动窗口**：单窗口 3-10 次 step 即可能 429。铺量前 ping 探测。
- **单代理串行**：不并发多代理（无加速且快速打穿配额）。
- **环境变量**：脚本已设 `PYTHONUTF8=1` + `PYTHONIOENCODING=utf-8`，不要改。
- **work_dir salvage**：脚本已用 tempdir，但 driver 在所有失败分支必须救回部分稿。
- **幂等 reuse**：driver kill/重启时若 `out` 已存在，直接搬运到 target 跳过 Kimi 调用。

### Codex（校验/诊断）

**调用模板**（单行位置参数 + 关 stdin + 硬超时）：
```bash
timeout 900 codex exec --sandbox read-only --ephemeral \
    -m gpt-5.4 \
    "<完整单行 prompt>" < /dev/null
```

**硬约束**：
- **必须 `< /dev/null`**：heredoc 多行 prompt 会让 Codex 等 stdin 卡死（04-15 教训）。
- **校验用 read-only**；批量修订用 `--sandbox workspace-write`。
- **目前未见配额**，可常驻使用。
- **产出短小高密**：30-100 行清单/工单，不要让它写长稿。

### Agent 编排

| 场景 | 谁来做 | 备注 |
|------|--------|------|
| Kimi 长任务（>22 分钟） | **主窗口 Bash run_in_background** | Agent 22 分钟 timeout 会连带 kill drive.py 子进程 |
| Codex 短任务（<10 分钟） | Agent 或主窗口均可 | |
| 多 Codex 并行（≤3） | 多 Agent | Codex 无配额限制 |
| 多 Kimi 并行 | **禁止** | 单代理串行铁律 |

## 四、决策点速查

### 已有传记 vs 新建档（工作流 A/B/C）

| 目标文件状态 | 走哪个工作流 | 关键理由 |
|-------------|------------|---------|
| 不存在 / 空壳 / 仅标题 | **A**（Kimi 整篇建档） | 没有可保护的旧资产 |
| 已存在有实质内容、需修订 | **C**（Codex 诊断 → 定点修订） | Kimi 整篇重写实测一手来源 -17.6% |
| 已存在合格、仅扩章节 | 局部追加，**不整篇覆盖** | 同上 |

### Kimi vs Codex vs opencode/glm 分工

| 任务 | 首选 | 次选 |
|------|------|------|
| 新传记长稿（300-650 行） | Kimi | opencode+glm-5.1（仅当 Kimi 配额耗尽） |
| 内部矛盾检测 / 工单产出 | Codex | — |
| 联网核对一手史料 | Codex（WebFetch wikisource/ctext/CBDB） | — |
| 跨文件一致性（BIO-005） | Codex（唯一可靠工具） | — |
| 按诊断单批量修订 | 主窗口 Edit 或 Codex `workspace-write` | opencode+glm |
| 机械审计（R1-R6 等） | `tools/*.py` 脚本 | — |

### content_filter 风险（Kimi）

**触发词**（实战累积）：斩、构陷、奏杀、阉党、下狱论死、瘐死、矫诏诛、棒杀

**规避方案**：
1. 用"本传客观叙事"风格，避免评价性动词与情绪化用语
2. 把狱案拆成多小节降密度（**钱龙锡范本**）
3. 政治暴力情节改用中性表述："被处死"→"卒于狱"、"构陷"→"上奏弹劾"
4. 4 连拒后**直接换 Codex 写本传**（钱龙锡 337 行成功范本）

### 主窗口 vs Agent 委派（与 R7 关系）

CLAUDE.md R7 要求执行委派 Agent，但**长任务（>22 分钟）必须主窗口 `run_in_background`**——R7 合规例外，已在 04-16 实战确认。

## 五、commit 前拦截阈值（4 条）

来自 `CLAUDE.md` 工作流决策树。任何对**已有传记**的修订 commit 前 Codex 必须做新旧对比评审，触发以下任一阈值即**禁止 commit**：

1. **一手古籍来源数下降 > 15%**
2. **《明史》独立卷号去重数大幅下降**（HEAD ≥ 3 但新版 ≤ 1，视同来源单薄化）
3. **上轮校验已修复的具体问题被回滚**（官职/年份/人名订正、存疑下沉等）
4. **引入新低学术源**（腾讯新闻/知乎/网易号/百家号等）

触发后必须做"旧版优质内容合并"补救（见 `P1_批次_20260415/` 与 `21份B类_20260416/` 的合并模板）。

## 六、跨工具陷阱

### `/tmp` 路径解释三义

| 工具 | `/tmp` 实际指向 |
|------|----------------|
| Git Bash | `E:\System_Temp\`（Cygwin/MSYS mount） |
| Claude Code Write 工具 | `G:\tmp\` |
| Python subprocess | `G:\tmp\` |

**规则**：跨工具传递路径**必须**用绝对路径或项目相对路径，禁用 `/tmp/xxx.md` 字符串。临时文件统一进项目内 `tmp/`。

### Windows GBK 控制台

- Kimi rich 渲染字符（• ◄ ► 等）pipe 给 GBK tee 必崩
- `kimi_safe_call.py` 已用"自写文件 + 搬运"规避
- 不要在 Bash 命令里 `kimi ... | tee log.txt`

### Agent 22 分钟 timeout

- 主窗口委派的 Agent 跑超 22 分钟会被 kill，连带 kill 子进程（drive.py）
- Kimi 单人长任务普遍 5-30 分钟，超时风险高
- **解法**：长任务用主窗口 Bash `run_in_background=true`，不委派 Agent

## 七、常见失败与 salvage

| 失败信号 | 排查与处理 |
|---------|-----------|
| `kimi_safe_call.py` 退出 429 | 5h 窗口耗尽。等窗口滑出（看上次成功调用的时间戳 +5h）；铺量前 ping 探测 |
| `kimi_safe_call.py` 退出 124 | 超时。把 `--timeout` 提高到 2400-3600，或拆 prompt |
| `kimi_safe_call.py` 退出 1 + stderr 含 `400 content_filter` | 改 prompt 风格（见第四章 content_filter 规避）；4 连拒后换 Codex 写本传 |
| `peer closed connection` / 网络中断 | salvage：去 `tmp/{批次}_out/` 找已有产物；若部分稿可用直接搬运 |
| Codex 卡死无产出 | 99% 是 stdin 未关闭。检查命令是否有 `< /dev/null`，是否有 heredoc 续行 |
| 产物落到错误目录（opencode/glm 路径固执） | 用脚本搬运，不要在 prompt 里加码对抗 |

**salvage 路径**：`kimi_safe_call.py` 在失败分支也保留了 work_dir（tempdir），driver 应在所有失败分支调 salvage 函数把部分稿搬到 `tmp/{批次}_out/`。张经 659 行、夏允彝 360 行均由此救回。
