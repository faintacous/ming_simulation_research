## 2026-04-12 08:28 — 人物名录建档 + 全库校验修订 + 工作流工具化

> 会话 `58bf2b93-1087-4f61-a0d0-78f54c99e2ea` | 时长约 33h 54m（跨 3 日）

### 摘要
从用户"收集人物信息"的一句话需求出发，完成了三日连续工作：67 人明朝人物传记建档 → 多轮 Codex 校验发现系统性质量问题 → 12 轮修订（单篇 + 全库规则）→ 工具化收尾（验收脚本集 + 证据链规范 + CLAUDE.md 更新）。期间走了多次弯路并做了根本性反思。

### 讨论过程

**第一阶段：建档（04-10 晚 ~ 04-11 中午）**
- 用户要求收集人物信息，先建名录再填充。确定 1550-1600 在世标准，12 类分类
- 主窗口启动 5 路 Claude 代理并行搜集 —— 用户立刻纠正："没有走 Kimi+Codex→Opus 流水线"
- 改用 Kimi 综合搜集，产出 `INDEX.md`（约 430 人）
- 用户选择先挑重点人物建档。第一批 17 人，Kimi 并发 17 路成功 10 人 + 接力重跑 5 人 + opencode 补 2 人
- 第二批起改用 opencode + glm-5.1，累计建了 62 份
- **关键纠正时刻**：用户注意到 opencode 轮次的文件普遍只有 100-200 行（vs Kimi 版 300-600 行），问"不是说要从网上搜的吗，怎么 prompt 里说基于训练知识？" —— 暴露出我把"避免 Explore Agent 卡死"过度推广成"禁用全部搜索工具"，导致后续 45 人都是 glm 凭训练知识编造
- 决定备份 51 份训练版到 `_backup_人物名录_training_only/`，然后用修正后的 prompt（允许 WebFetch，只禁 Agent）试跑 2 人验证，成功后批量重做
- 重做过程又遭遇连串问题：glm 默认模型 glm-4.6 有 SSE 超时、`question` 工具禁不掉导致死锁（备份文件夹在 `人物名录/` 内被模型读到，触发澄清问题）、模型固执地把文件写到 `人物档案/` `历史人物/` `05-军事与边防/` 等"更合理"的位置
- 解决：把备份移出 `人物名录/`、agent 文件显式指定 `model: zhipuai-coding-plan/glm-5.1`、在 Python 脚本里加 `find_and_move_file` 自动扫描搬运（放弃对抗模型路径固执）
- 最终 66 人齐活（小西行长因 3 次卡死只拿到 34 行 stub，后续重写到 257 行）

**第二阶段：校验（04-11 晚）**
- 用户提出"用 codex 对文档进行一下校验"
- 我最初建议 20 人独立校验。Codex 在张居正一篇校验中顺带看了 6 个文件，暴露效率：改为"一次调用校验 20 人"综合报告
- 综合报告 40 行精华，发现系统性问题：**争议说法定论化、同文自相矛盾、野史进入主叙述**。具体指出 6 条严重（张居正 1622 < 1630 时间倒置、海瑞族属+一条鞭法双重矛盾、李时珍蓬溪知县实为其子、王喜姐/郑贵妃野史进入正文、朱翊钧戏剧化细节）
- 用户追问："Codex 本身没法完成史料核实吗？" —— 我最初以为 Codex 只能找矛盾不能查史料，用户质疑后我重新检查，发现 Codex 第一轮报告里其实访问了 wikisource 做定点抽查。于是跑第三轮系统联网核对
- 第三轮 Codex 真的访问 wikisource/ctext/CBDB，对 13 个可疑点每条附带一手史料证据。例如《明史·李时珍传》卷299 原文"时珍官楚王府奉祠正，子建中，四川蓬谿知县"直接证伪"李时珍任蓬溪知县"说
- 用户确认"按照建议"：两路并行（A 修 6 条硬错误 + B Codex 联网核对）→ A2 修 4 条新硬错误（张居正西藏/麻贵回族/李成梁义子/陈矩逸闻）→ 全库工单 C → BIO-001~009 规则化
- 用户要求 5/6/7 并行：Phase 1 修剩余问题（BIO-005 跨文件/BIO-006 严重/5 份来源汇总）→ Phase 2 工具化/证据链/CLAUDE.md

**第三阶段：收尾（04-12 早）**
- BIO-005 跨文件一致性检查暴露单文件修订盲区：朱翊钧.md 已经降级"十岁太子"典故，但冯保.md 和高拱.md 还在写定论 —— 跨文件的同事件叙述需要专门的 BIO-005 环节，单文件修订不够
- BIO-006 六规则审计整体合规率 86.6%，发现 3 份严重不合规（小西行长截断、紫柏真可来源汇总截断、憨山德清 R2 搬运未到位）
- Phase 2 三路完成：工具化（`_validation/tools/` 5 个脚本 + 新传记验收流程.md）、证据链规范（E1-E6 六级 + BIO-007/008/009 三条新规则）、CLAUDE.md 更新（多 Agent 表格重构 + 工具能力边界 + 推荐工作流 + 六个教训）
- 收尾前用户指出 `docs/devlog.md` 现有内容"看起来不像本项目" —— 实际是 03-29 的府域设计研究记录，确实是本项目，但 AI 生成风格太重

### 关键决策

- **Kimi 走流水线，opencode 作备用**：Kimi 配额耗尽后需要备用生成器。确定 opencode + glm-5.1（显式指定模型）作为 Plan B，用 agent 配置文件禁用 read/task/list/glob 等工具
- **放弃对抗模型的路径固执**：与其在 prompt 里反复加"必须写到绝对路径 X"（无效），不如让 Python 脚本每次运行后扫描全项目找包含人物名的 md 文件自动搬运。这个决策抢救了约 20 个错位文件
- **接受 Codex 的"严重/中等/一般"分级**：Codex 自己用的分级格式比我设计的 ✅/⚠️/❌ 模板更好，直接采用
- **一次性综合校验 > 逐人独立校验**：Codex 天然擅长跨文件对比，20 人一次调用比 20 次独立调用高效得多
- **BIO-005 作为独立环节**：单文件修订会留跨文件矛盾（朱翊钧修了但冯保/高拱没同步）。必须在工作流里加一道"跨文件一致性检查"
- **证据链用 E1-E6 六级而非简单二分**：研究环境里内容有天然的层次差异，一刀切"可信/不可信"会误伤。六级允许渐进迁移、并存兼容
- **工具显式声明 model**：glm-4.6 SSE 超时教训后，agent 配置文件必须显式写 `model:` 字段

### 否决/放弃的方案

- **Claude 代理替代 Kimi 做搜集**：第一轮启动 5 路 Claude 代理，被用户立刻否决，改用 Kimi 流水线
- **opencode prompt 禁用全部搜索工具**：为了避免 Explore Agent 卡死而禁用所有 Glob/Read/ls/WebFetch/Agent，结果误伤 WebFetch 导致 45 人是训练知识版。正确做法是只禁 Agent，保留 WebFetch
- **在 prompt 里反复强调绝对路径**：模型固执不听，白费 token。改用脚本搬运
- **BIO-002 全库逐条引用打 6 级标签**：太庞杂，改为轻量版"识别高风险文件并加顶部警示块"
- **BIO-004 全库机械重写时间**：改为"只修硬错误（第 1 类年号换算错），其余列入待审"

### 未完成
